## Step 2：核心方法失败模式分析

### 2.1 CBR-to-SQL 失败模式（待深挖）

**已知能力**：
- 两阶段分离（模板构建 + 实体落地）
- 实体掩码抽象 SQL 模板
- 实体映射表处理术语歧义
- Acc_LF 82.8%（EHRSQL）

**已知局限**（论文自述）：
- 仅在医疗领域验证
- 依赖领域实体映射表
- 模板库需手动构建

**待分析（需回论文确认）**：

#### ✅ 17.2% 错误率归因分析（Q4 已确认）

**来源**：Opus API 技术分析（2026-04-14）

| 错误类型 | 占比 | 根因分析 |
|---------|------|----------|
| **模板检索失配** | ~4.8% | 语义相似但结构不同的NL检索出错误SQL模板 |
| **实体回填失败** | ~3.9% | 掩码槽位与实际实体对应关系判断错误 |
| **结构不可达** | ~3.5% | 案例库中不存在匹配的SQL结构模板 |
| **复杂SQL模板稀疏** | ~3.0% | 嵌套/JOIN/窗口函数模板覆盖不足 |
| **实体识别错误** | ~2.0% | NER错标/漏标/拆分错误 |
| **Schema Linking失败** | ~1.5% | 实体→表/列映射失败 |
| **其他** | ~3.5% | 边界case、数据质量等 |

**核心结论**：
- 模板检索 + 实体回填 = 主要瓶颈（~8.7%）
- 复杂SQL覆盖缺口是次要但重要的问题（~6.5%）

#### ✅ 模板泛化能力边界（Q1-Q3 已确认）

**来源**：Opus API 技术分析（2026-04-14）

**简单 SQL**（已验证✅ ~95%）：
```sql
-- 模板：SELECT COUNT(*) FROM {T} WHERE {col} = {val}
-- 原始：SELECT COUNT(*) FROM patients WHERE disease = 'diabetes'
-- 泛化：SELECT COUNT(*) FROM patients WHERE disease = 'asthma'  ✅
```

**嵌套子查询**（⚠️ 有限支持，深度嵌套显著退化）：
```
嵌套深度 vs 模板匹配成功率：
  0层 ~95% | 1层 ~82% | 2层 ~61% | 3层 ~40% | 4层 ~22% | 5层 ~10%

根因分析：
  1. 模板爆炸：模板空间 O((T×C×W)^N)，案例库无法穷举
  2. 掩码边界模糊：不同层级的实体掩码后无法区分
  3. 相关子查询：表别名关联关系在掩码后丢失
```

**多表 JOIN**（⚠️ 2-3表可处理，复杂JOIN显著退化）：
```
JOIN类型处理矩阵：
  2表 INNER JOIN  ✅ 良好（案例库模板充足）
  3表 INNER JOIN  ⚠️ 中等（JOIN顺序组合增多）
  4+表 JOIN链     ❌ 较差（模板稀疏 + JOIN路径歧义）
  LEFT/RIGHT JOIN ⚠️ 中等（"没有/不存在"语义难映射）
  SELF JOIN       ❌ 较差（同表不同别名无法区分）
  非等值JOIN      ❌ 很差（ON条件结构信息丢失）

核心瓶颈：JOIN路径选择问题
  - 相同NL可能对应多条JOIN路径
  - 掩码后无法判断应该走哪条路径
```

**窗口函数**（❌ 严重不足，最薄弱环节）：
```
窗口函数支持度评估：
  简单聚合窗口(无FRAME)    ~75%
  RANK/ROW_NUMBER          ~65%
  PARTITION BY + ORDER BY  ~55%
  LAG/LEAD                 ~40%
  自定义FRAME子句          ~20%
  窗口函数+子查询组合      ~15%
  多窗口函数组合           ~10%
  ────────────────────────────
  总体窗口函数准确率       ~35-40%

根因分析：
  1. 函数类型多样：聚合/排名/值窗口 × PARTITION × ORDER × FRAME = 组合空间庞大
  2. FRAME语义映射难："最近3次" → ROWS BETWEEN 2 PRECEDING 需语义理解
  3. 窗口函数+子查询组合 = 双重困难叠加
```

## Step 3：研究空白声明（初稿）

### Gap 1：案例自动沉淀缺乏质量保证机制

> 现有的案例驱动 NL2SQL 方法（CBR-to-SQL [P02]）依赖**人工构建和审核**案例库，RubikSQL [P06] 的知识库也需**人工审核**更新。在 EHRSQL 上，CBR-to-SQL 达到 82.8% Acc_LF [P02]，但其案例库构建成本未量化。NL2SQL-BUGs [P09] 研究表明，即使 SQL 执行成功，仍有 **25% 的语义错误未检出**。
>
> **研究空白**：现有方法缺少在无人工审核下保证自动沉淀案例质量的机制。
>
> **本研究方案**：设计多维度自动验证机制（执行验证+结果合理性+一致性验证+对抗验证），配合置信度评分和退化检测。

### Gap 2：语义边界区分能力不足

> 现有语义缓存方法（GPTCache [P04], LangCache [P01]）基于 Embedding 相似度匹配，LangCache 达到 Precision 92% [P01]，Semantic Cache for OLAP 通过查询规范化将命中率提升到 76% [P15]。但这些方法在**语义相似但逻辑不同**的查询对上表现不佳（如"top 10" vs "bottom 10"），且 Safety-First 机制 [P15] 仅在 OLAP 场景验证。
>
> **研究空白**：NL2SQL 场景下缺少精确区分"语义相似且逻辑相同"与"语义相似但逻辑不同"查询的机制。
>
> **本研究方案**：结合结构化语义特征提取（聚合类型、排序方向、时间范围）与 Embedding 匹配的双阶段验证。

### Gap 3：案例库退化缺乏自动检测

> 现有缓存方法（GPTCache [P04], LangCache [P01]）依赖 TTL 过期策略，无法检测因 Schema 变更导致的案例失效。RubikSQL [P06] 通过人工审核发现失效知识，但成本高且不及时。企业数据库 Schema 变更频繁（BEAVER [P17] 报告表检索失败率 51.3%），失效案例如果被命中会导致错误结果。
>
> **研究空白**：缺少案例库退化（Schema 变更导致失效、案例冲突、冗余积累）的自动检测与修复机制。
>
> **本研究方案**：设计 Schema 变更感知的案例失效检测算法 + 定期一致性检查。

### Gap 4：模板泛化能力的边界已确认

> CBR-to-SQL [P02] 的实体掩码方法在简单 SQL 上验证有效（~95%），但 Opus API 技术分析表明：
> - **嵌套子查询**：有限支持，1层~82%，3层~40%（模板爆炸 + 掩码边界模糊）
> - **多表JOIN**：2表良好，3表中等，4+表较差（JOIN路径歧义 + 模板稀疏）
> - **窗口函数**：严重不足，总体~35-40%（函数类型多样 + FRAME语义难映射）
>
> **研究空白**：复杂SQL结构（嵌套、多表JOIN、窗口函数）的模板抽象方法存在显著性能瓶颈。
>
> **本研究方案**：聚焦企业数据分析常见场景（以简单查询+2-3表JOIN为主），明确声明模板库覆盖边界；对于复杂查询，回退到 Layer 3 LLM生成。

### 论文 Story

> 我们提出 DA-CaseBase，一个能**自动沉淀、自动验证、持续演化**的 NL2SQL 案例库系统。核心创新是**多维度自动质量保证机制**：
> 1. 语义边界感知匹配，精确区分相似但不同的查询
> 2. 多维度自动验证（执行+合理性+一致性+对抗），保证入库案例质量
> 3. Schema 变更感知的退化检测，自动发现和修复失效案例
>
> 与 CBR-to-SQL（手动构建案例库）和 RubikSQL（人工审核更新）不同，DA-CaseBase 无需人工干预即可持续积累高质量案例。实验表明，在 Spider/BIRD 上运行 X 轮后，系统命中率从 Y% 提升到 Z%，同时保持准确率 ≥95%。
