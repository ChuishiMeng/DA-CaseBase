## 一、PDF核心内容总结

### 1. 案例匹配的四阶段目标

| 阶段 | 目标 | 典型问题 | 解决手段 |
|------|------|----------|----------|
| **第一阶段：变量变化** | 参数标准化 | "今年产值最高" → "去年产值最高" | 槽位定义、参数抽取、标准化 |
| **第二阶段：语序变化** | 顺序弱敏感 | "销售总金额在2023年是多少" → "2023年销售总金额是多少" | 语义片段拆解、成分重组 |
| **第三阶段：关键词变化** | 同义表达归一 | "销售额" vs "卖了多少钱" | 业务术语归一、examples覆盖 |
| **第四阶段：口语化** | 缩写/省略补全 | "23年一共卖了多少钱" | 口语规范化、隐式语义补全 |

### 2. 当前方法三大问题

| 问题 | 表现 | 影响 |
|------|------|------|
| **问题表示不稳定** | 逐案生成roles，不同候选案例生成不同结构 | 匹配结果依赖候选案例上下文，而非问题本身 |
| **整体语义筛选没用examples** | 用case_name而非examples集 | 标题写法影响召回，逻辑接近但标题不同易误伤 |
| **匹配与提参不共享骨架** | 匹配用roles，提参重新组合 | 命中依据和提参依据不一致，难以解释 |

### 3. 改进方案四步重构

**Step 1: 参数识别和验证**
- 提取可变部分（时间、组织、区域、TopN）
- 参数标准化（缩写→正式表达）
- 实体类参数校验（能否落到真实业务对象）

**Step 2: 词组归一化**
- 拆解时间、指标、主体、限定条件
- 同义表达映射到规范说法
- 不急于强行归一，保留多种可能

**Step 3: 问题补全**
- 受控补全（有明确依据的内容）
- 缩写还原（23年→2023年）
- 口语规范化（卖了多少钱→销售总金额）
- 歧义保留为待确认状态

**Step 4: 案例比较**
- 使用前三步收敛后的结果
- 先比较主干→再比较参数→最后分层判断
- 匹配与提参共享同一份问题理解

### 4. 案例表示改进（以Schema为准）

**关键思想**：
- 槽位不是抽象名字，必须绑定到schema字段
- 槽位能接受什么值，必须有明确约束
- 问题表示应以schema中真实存在的表和字段为准

**示例**：
```
用户问题: 查询今年华东区域销售额最高的前10个项目
规范表达: 查询[stat_date][region]销售额最高的前10个项目

槽位绑定:
- 今年 → stat_date字段
- 华东区域 → region_name字段
- 销售额最高 → ORDER BY sales_amount_sum DESC
- 前10个 → LIMIT {top_n}
- 项目 → project_name字段
```

## 三、工程→学术转化路径

### 1. 案例表示形式化

**PDF工程方案**：
```
案例结构 = 问题规范表达 + SQL模板 + 槽位绑定schema
槽位定义 = 字段名 + 值域来源 + 标准化值
```

**P3学术转化**：
```
案例定义 C = (Q_norm, S_template, B_schema, M_meta)
其中:
- Q_norm: 规范化问题表示
- S_template: SQL模板（多粒度抽象）
- B_schema: Schema绑定映射
- M_meta: 元信息（上下文层+质量层+关联层）
```

### 2. 匹配算法形式化

**PDF工程流程**：
```
Step1: 参数识别 → Step2: 词组归一 → Step3: 问题补全 → Step4: 案例比较
```

**P3学术转化**：
```
Match(Q, C) = 
  Let Q' = Normalize(Q)       // Step1-3合并：规范化
  Let C_candidates = Embed(Q') // 粗筛：Embedding召回
  For each C in C_candidates:
    If StructMatch(Q', C) → exact_match    // Step4：结构化精确匹配
    Else If SemMatch(Q', C) → fuzzy_match   // 语义相似匹配
  Return best_match

其中:
- Normalize = ParameterExtract + TermNormalize + QuestionComplete
- StructMatch = 主干比较 + 参数范围比较 + 分层判断
- SemMatch = Embedding相似度（阈值）
```

### 3. 需量化的指标（实验设计方向）

| 指标 | PDF未提供 | P3需实验确定 |
|------|----------|-------------|
| 各阶段命中率贡献 | ❌ | 需分阶段ablation实验 |
| 参数识别准确率 | ❌ | 需定义准确率指标 |
| 词组归一覆盖率 | ❌ | 需统计同义表达覆盖率 |
| 口语补全成功率 | ❌ | 需人工标注验证 |
| 整体匹配Precision | ❌ | 目标 ≥95% |

## 四、缺失点 → 研究空白确认

PDF未涉及的三大问题，正是P3的核心研究空白：

### 1. 质量保证机制（Gap 4核心）

**PDF现状**：无验证机制，依赖人工或无控制
**P3创新**：多维度自动验证
- 执行验证：SQL能执行
- 结果验证：结果合理性检查
- 一致性验证：相似问题的结果一致性
- 对抗验证：微扰问题是否得到一致结果

### 2. 退化检测（Gap 4核心）

**PDF现状**：无检测机制
**P3创新**：Schema变更感知 + 定期一致性检查
- Schema变更检测：表/字段增删改
- 案例失效检测：SQL执行失败
- 冗余/冲突检测：相似案例合并

### 3. 自动沉淀（Gap 4核心）

**PDF现状**：无入库流程说明
**P3创新**：置信度评分驱动的生命周期管理
- 新增 → 验证 → 成熟 → 退化 → 淘汰
