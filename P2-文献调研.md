## 四、关键技术发现

### 4.1 语义相似度匹配 (RQ1)

#### 核心方法对比

| 方法 | 论文 | 命中率 | 适用场景 |
|------|------|--------|---------|
| **Embedding 相似度** | GPTCache | 60-80% | 问答场景 |
| **领域微调 Embedding** | LangCache | 90%+ | 特定领域 |
| **LLM 查询规范化** | P15-Semantic-Cache | 76% | OLAP 分析 |

#### 关键发现

1. **小模型 + 领域微调可超越大模型** (P01-LangCache)
   - 149M 参数 ModernBERT 微调后 Precision 达 92%
   - 超越 7B 参数的 gte-Qwen2

2. **查询规范化比直接 Embedding 更准确** (P15)
   - 命中率从 28%（Text Cache）提升到 76%
   - Safety-First 机制保证精确率

3. **Data Agent 场景不适用答案缓存**
   - 数据实时变化，答案必须重新查询
   - 语义匹配应用于：**检索相似问题作为 Few-shot 示例**

### 4.2 案例模板表示与泛化 (RQ2)

#### 核心方法：两阶段分离

```
CBR-to-SQL (P02): 模板构建 + 实体落地

用户问题: "哮喘患者用过布地奈德"
    ↓ Step 1: 模板构建
SQL 模板: SELECT COUNT(*) FROM {table} WHERE {disease_col} = {disease_val} AND {drug_col} = {drug_val}
    ↓ Step 2: 实体落地
实体映射: 哮喘 → diagnoses.icd9_code = '493', 布地奈德 → medications.drug_name = 'Budesonide'
    ↓
最终 SQL: SELECT COUNT(*) FROM diagnoses d JOIN medications m ON d.patient_id = m.patient_id WHERE d.icd9_code = '493' AND m.drug_name = 'Budesonide'
```

#### 关键发现

1. **实体掩码是模板抽象的核心** (P02)
   - 将具体值替换为占位符 {disease}, {drug}
   - 相同结构的查询映射到同一模板

2. **实体映射表解决术语歧义** (P02)
   - 业务术语 → 数据库字段映射
   - 支持 LLM 消歧

3. **TailorSQL 方法不适用本研究** (P03)
   - 依赖 SQL 历史查询数据（本研究不使用）
   - 核心价值：Document Store 设计模式可借鉴

### 4.3 三层架构设计 (RQ3)

#### RubikSQL 四阶段工作流 (P06)

```
Stage 1: Database Context Engineering
    - Schema 增强（添加业务描述）
    - 常用 Join 路径标注

Stage 2: User Query Augmentation
    - 意图识别
    - 实体链接
    - 上下文注入

Stage 3: Knowledge Base Indexing
    - 检索相关文档
    - 检索相似案例

Stage 4: Knowledge Distillation
    - SQL 生成
    - 执行验证
```
